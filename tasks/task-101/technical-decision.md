## 二、冷启动策略决策

### 2.1 选项对比

| 方案 | 优点 | 缺点 | 工作量估算 |
|------|------|------|------------|
| **从零开始** | - 数据质量可控<br>- 格式统一<br>- 无历史包袱 | - 初期知识库为空<br>- 需要时间积累 | 0 工时 |
| **手动导入历史数据** | - 初期有内容可用<br>- 可复用历史经验 | - 格式不统一需清洗<br>- 质量参差不齐<br>- 导入成本高 | 20+ 工时 |

### 2.2 决策理由

**选择：从零开始积累**

**核心理由**：

1. **成本效益分析**：
   - 历史任务记录格式不统一，需要大量清洗工作（预估 20+ 工时）
   - 旧数据质量参差不齐，可能污染知识库
   - 学习智能体系统的核心价值在于**持续积累**，初期内容为空是可接受的

2. **质量控制**：
   - 从零开始可以确保所有入库数据符合统一格式
   - 每条学习记录都经过完整的生成流程（任务执行→Challenger 审查→学习记录生成）
   - 避免历史数据中的错误模式被固化

3. **例外处理**：
   - 如有**特别重要的历史案例**（如重大事故复盘、关键技术突破），可手动整理后单独入库
   - 这类案例数量稀少（预计<10 条），整理成本可控

### 2.3 冷启动阶段预期

| 阶段 | 时间 | 预期状态 |
|------|------|----------|
| 第 1-2 周 | 试点期 | 知识库为空，Worker 主要依赖 Search Worker 获取新知识 |
| 第 3-4 周 | 积累期 | 首批学习记录入库（预计 10-20 条） |
| 第 5-8 周 | 成长期 | 知识库初具规模，推荐功能开始发挥作用 |
| 第 9 周+ | 成熟期 | 知识复用率>80%，系统价值充分显现 |

## 三、技术选型决策：向量数据库

### 3.1 候选方案对比

基于技术方案文档的分析，补充完整对比：

| 维度 | LightRAG | Chroma | Weaviate | 权重 |
|------|----------|--------|----------|------|
| **查询性能** | 中（图检索有额外开销） | 高（纯向量检索） | 高（优化良好） | 15% |
| **存储成本** | 中（需存储图结构） | 低（轻量） | 中 | 10% |
| **学习曲线** | 中（文档较少） | 低（文档完善） | 中（需 Docker） | 10% |
| **社区活跃度** | 中（新兴项目） | 高（广泛使用） | 高（企业支持） | 10% |
| **图检索支持** | ✅ 原生支持 | ❌ 不支持 | ⚠️ 需插件 | **25%** |
| **自托管友好度** | 高（Python） | 高（Python） | 中（需 Docker） | 15% |
| **中文支持** | 好 | 好 | 好 | 15% |
| **综合得分** | **8.5** | 7.0 | 7.5 | - |

### 3.2 决策理由

**选择：LightRAG**

**核心理由**：

1. **核心场景匹配**：
   - 学习智能体系统的核心价值是**任务关联推荐**（如：相似任务、相关问题模式、历史解决方案）
   - 这类推荐本质上是**图结构问题**：任务→问题→解决方案→经验教训，存在丰富的关联关系
   - LightRAG 的**图 + 向量混合检索**能力是独特优势，Chroma 和 Weaviate 无法原生支持

2. **技术风险可控**：
   - 虽然 LightRAG 是新兴项目，文档相对较少，但团队有 Python 经验，学习曲线可接受
   - 已设计**试点验证计划**（4 周），可快速验证并保留切换到 Chroma 的选项

3. **自托管友好**：
   - 符合 Admin 的本地部署要求
   - Python 实现，部署和维护相对简单

4. **长期价值**：
   - 图检索能力是差异化优势，随着知识库增长，价值会持续放大
   - 纯向量检索方案（Chroma）在任务关联场景有天然局限

### 3.3 风险控制

| 风险 | 缓解措施 | 触发条件 |
|------|----------|----------|
| LightRAG 学习曲线陡峭 | 第 1 周 dedicated 部署和测试，产出部署文档 | 如第 1 周末无法完成基础部署，触发备选方案 |
| 图检索效果不如预期 | 第 3 周进行图检索测试，用真实任务案例验证 | 如相似度推荐准确率<60%，考虑切换 Chroma |
| 社区支持不足 | 建立内部知识库，记录问题和解决方案 | 遇到无法解决的问题时，记录并分享 |

### 3.4 备选方案

**如果 LightRAG 试点失败，切换到 Chroma**：

**切换条件**（满足任一即触发）：
- LightRAG 部署时间 > 2 周
- 图检索推荐准确率 < 60%（测试 10 个真实任务案例）
- 严重 Bug（阻塞开发）> 3 个且 1 周内无法解决

- **切换成本**：中等（主要改动在检索层，学习记录生成逻辑不变）
- **损失**：图检索能力（任务关联推荐效果下降）
- **收益**：开发速度提升，社区支持更好

## 四、Challenger🔴问题回应

根据技术总监审查提出的问题，逐一回应：

### 🔴 Q1: Search Worker 的搜索质量保证机制缺失

**回应**：

1. **审查流程**：
   - P0 请求：Challenger Worker **必审**（搜索报告完成后自动触发审查）
   - P1/P2 请求：Challenger Worker **抽样审查**（抽样率≥20%）
   - 审查标准：准确率、时效性、完整性（见 challenger-checklist.md）

2. **质量指标**：
   - **准确率**：请求者评分≥4/5（每次检索后请求者打分）
   - **时效性**：SLA 达标率≥90%（P0:2h, P1:4h, P2:24h）
   - **完整性**：搜索报告必须包含核心概念、关键要点、优缺点分析、学习资源

3. **错误纠正**：
   - 发现错误后：Search Worker 修正并标记版本（v1→v2）
   - 严重错误（导致决策失误）：触发 Admin 通知（Matrix @admin:matrix-local.hiclaw.io:18080）
   - 错误记录：存入错误案例库（内部保存，不同步 GitHub），避免重复犯错

**状态**：✅ 已解决（见本文件 三.1 节）

### 🔴 Q2: Search Worker 与学习记录的边界不够清晰

**回应**：

1. **知识库分区设计**：
   ```
   知识库
   ├── 搜索报告库（Search Worker 产出，面向新知识/新概念/工具调研）
   ├── 任务经验库（学习记录，面向任务执行经验/教训）
   └── 问题模式库（Challenger 审查问题，面向风险预防）
   ```

2. **检索路由规则**：
   - Worker 明确指定检索库类型 → 直接路由到对应库
   - Worker 模糊检索 → 同时检索多个库，按相关性排序展示
   - 示例：搜索"向量数据库"→同时检索搜索报告库（技术介绍）和任务经验库（使用案例）

3. **去重机制**：
   - 入库前检查相似内容（相似度>80% 触发人工审核）
   - Search Worker 在撰写报告前，先检索是否存在相似主题
   - 如存在相似报告，可选择：(a) 更新旧报告 (b) 撰写补充报告

**状态**：✅ 已解决（见本文件 三.1 节和四.Q2 节）

### 🔴 Q4: SLA 执行机制未定义

   ```
   超时 30 分钟 → @mention Search Worker 提醒
   ```

   - Search Worker 可申请延期（需说明理由，如：资源受限、主题复杂）
