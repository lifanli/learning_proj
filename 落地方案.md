## 三、工作流程

### 3.1 标准审查流程

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1: Worker 提交工作成果                                  │
│ 位置：shared/tasks/{task-id}/                               │
│ 内容：plan.md, 实现文件，自测报告等                          │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2: Manager 通知反问 Worker                              │
│ 方式：@mention @review-worker                               │
│ 内容：任务链接 + 审查重点（可选）                            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3: 反问 Worker 审查并提交问题（<2 小时）                 │
│ 输出：review-questions.md                                   │
│ 内容：总体评价 + 3-7 个问题 + 风险识别 + 建议                │
│ 通知：@mention Worker 和 Manager                            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 4: Worker 回答问题（时限由反问 Worker 设定）             │
│ 输出：review-responses.md                                   │
│ 要求：逐一回答，提供证据/数据支撑                            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 5: 反问 Worker 评估                                     │
│ 通过 → 任务继续推进                                         │
│ 不通过 → 跟进问题（最多 1 轮）                               │
│ 输出：review-final.md                                       │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 6: Manager 最终确认                                     │
│ 审查问答记录，确认任务可以进入下一阶段                       │
└─────────────────────────────────────────────────────────────┘
```

## 五、任务分类与审查深度

### 5.1 任务分级

| 级别 | 描述 | 审查要求 | 问题数量 | 追问轮次 |
|------|------|----------|----------|----------|
| **S 级** | 核心功能、高风险 | 强制审查 | 5-7 个 | 2 轮 |
| **A 级** | 重要功能、中等风险 | 强制审查 | 3-5 个 | 1 轮 |
| **B 级** | 常规功能、低风险 | 可选审查 | 2-3 个 | 1 轮 |
| **C 级** | 小修复、文档更新 | 免审查 | - | - |

### 5.2 分级标准

**S 级（关键）**:
- 核心业务逻辑
- 涉及用户资金/隐私
- 影响面广（>50% 用户）
- 新技术/新架构首次使用

**A 级（重要）**:
- 主要功能模块
- 涉及外部集成
- 性能敏感场景

**B 级（常规）**:
- 常规功能开发
- UI 调整
- 已知模式的实现

**C 级（简单）**:
- Bug 修复
- 文档更新
- 配置修改

## 二、问题列表

### 问题 1: [安全]
**问题**: 方案中提到使用 JWT 存储 token，具体存储在哪里？localStorage 还是 sessionStorage？有没有考虑 XSS 攻击的风险？  
**原因**: token 存储方式直接影响安全性，localStorage 易受 XSS 攻击  
**建议思考方向**: 是否需要使用 httpOnly cookie？或者加密存储？

### 问题 2: [边界场景]
**问题**: 当用户提交登录后，如果后端 API 超时或返回 500 错误，前端如何展示？有重试机制吗？  
**原因**: 网络不稳定是常见场景，需要优雅降级  
**建议思考方向**: 错误提示文案、重试按钮、超时时间设置

### 问题 3: [性能]
**问题**: 登录页面是否做了代码分割？首屏加载时间目标是多少？  
**原因**: 登录是用户第一个接触的页面，加载速度影响第一印象  
**建议思考方向**: 懒加载、预加载、资源优化
