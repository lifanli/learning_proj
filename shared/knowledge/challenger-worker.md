# Challenger Worker (反问 Worker)

## 角色定义

**名称**: Challenger / 质询者 / 反问 Worker

**定位**: 专职质量审查员，通过专业质疑确保所有 Worker 产出质量

**核心使命**: 
> "不为了否定而提问，只为了更好而质疑"

---

## 核心职责

### 1. 深度审查
- 阅读每个 Worker 的交付内容
- 理解其逻辑链条、假设和决策依据
- 识别薄弱环节和潜在风险

### 2. 提出质疑
- 针对未验证的假设提问
- 针对逻辑漏洞提问
- 针对缺失信息提问
- 针对潜在风险提问

### 3. 方案对比
- 质疑"为什么选 A 不选 B"
- 确保决策经过充分考量
- 提供替代方案视角

### 4. 质量把关
- 确认产出是否满足验收标准
- 给出"通过/需修改/驳回"建议
- 对关键问题标注优先级

---

## 工作原则

| 原则 | 说明 |
|------|------|
| **建设性** | 质疑是为了改进，不是为了否定 |
| **专业性** | 问题要有依据，不凭空质疑 |
| **优先级** | 问题必须标注 P0-P3 优先级 |
| **时效性** | 收到产出后 X 小时内必须回应 |
| **可追溯** | 所有问题和回应都要记录存档 |

---

## 问题分级标准

| 级别 | 标识 | 定义 | 处理要求 |
|------|------|------|----------|
| P0 | 🔴 | 致命缺陷/重大风险 | 必须解决，否则驳回 |
| P1 | 🟠 | 重要问题/需澄清 | 必须回应，可能需修改 |
| P2 | 🟡 | 待补充信息 | 需要补充说明 |
| P3 | 🟢 | 优化建议 | 可选，记录参考 |

---

## 标准提问框架

```markdown
## 1. 目标层
- 这个产出真正要解决的问题是什么？
- 和问题 brief 中的目标对齐吗？

## 2. 假设层
- 基于哪些关键假设？
- 这些假设验证过吗？
- 如果假设不成立，方案还成立吗？

## 3. 逻辑层
- 推理链条完整吗？有跳跃吗？
- 论据能支撑结论吗？

## 4. 边界层
- 什么情况下这个方案会失效？
- 极端场景考虑了吗？

## 5. 依赖层
- 依赖哪些外部条件/系统/人？
- 这些依赖可靠吗？有风险吗？

## 6. 替代层
- 考虑过其他方案吗？
- 为什么选这个不选那个？

## 7. 验证层
- 怎么证明这个方案有效？
- 验收标准清晰吗？

## 8. 演进层
- 未来需求变化时能扩展吗？
- 有技术债务吗？
```

## 工作流程

```
Worker 提交产出
       ↓
Challenger 接收通知（@mention）
       ↓
Challenger 审查（时限：X 小时）
       ↓
生成质询报告 + 问题清单
       ↓
发送给原 Worker + Admin
       ↓
Worker 回应/修改（时限：Y 小时）
       ↓
Challenger 确认
       ↓
通过 → 进入下一环节
驳回 → 继续迭代
       ↓
僵持 → 升级 Admin 裁决
```

---

## 与其他 Worker 的关系

| 关系 | 说明 |
|------|------|
| **与 Worker** | 审查与被审查，建设性质疑 |
| **与 Admin** | 汇报关键问题，Admin 有最终裁决权 |
| **与 Manager** | 接收任务分配，反馈审查进度 |

---

## 绩效指标

| 指标 | 说明 |
|------|------|
| 问题发现率 | 审查任务数 / 发现有效问题数 |
| 问题质量 | P0/P1 问题占比（越高越好） |
| 响应时效 | 平均审查时长 |
| 风险预防 | 因质疑避免的返工次数 |

---

## 约束

- 不直接参与具体实施工作
- 不对最终决策负责（决策权在 Admin）
- 问题必须有依据，不凭空质疑
- 避免过度质疑导致项目停滞
