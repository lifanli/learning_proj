## 一、核心概念

### 1.1 什么是反问 Worker？

**反问 Worker** 是一个独立的 AI 角色，专门负责：
- 审查其他 Worker 的工作输出
- 提出专业、有深度的问题
- 促使 Worker 更深入思考和完善方案
- 识别盲点、风险和未考虑的场景

### 1.2 与现有角色的关系

```
┌─────────────────────────────────────────────────────────┐
│                    人类管理员                            │
│                    (最终决策者)                          │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                    Manager                               │
│              (任务分配 + 协调)                           │
└─────────────────────────────────────────────────────────┘
                          ↓
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│  前端工程师   │ │  后端工程师   │ │  反问 Worker  │
│  (执行角色)   │ │  (执行角色)   │ │  (审查角色)   │
└───────────────┘ └───────────────┘ └───────────────┘
        ↑                 ↑                 ↑
```

## 二、反问 Worker 的核心职责

### 2.1 审查范围

| 审查对象 | 审查内容 | 提问重点 |
|----------|----------|----------|
| **前端工程师** | UI 设计、技术方案、代码实现 | 用户体验、性能、可维护性、边界场景 |
| **后端工程师** | API 设计、数据库方案、架构 | 安全性、扩展性、一致性、容错 |
| **产品经理** | 需求文档、功能设计 | 用户价值、优先级、可行性 |
| **测试工程师** | 测试计划、用例设计 | 覆盖率、边缘场景、自动化程度 |

### 2.2 问题质量要求

反问 Worker 提出的问题必须满足：

1. **具体而非笼统**
   - ❌ "这个方案有风险吗？"
   - ✅ "当用户并发量超过 1000 时，这个缓存策略会怎样？"

2. **有建设性而非纯批判**
   - ❌ "这个设计太复杂了"
   - ✅ "这个设计引入了 3 个新依赖，有没有更轻量的替代方案？"

3. **基于事实而非假设**
   - ❌ "我觉得这样不行"
   - ✅ "根据 XX 文档，这个 API 在移动端有兼容性限制，考虑过吗？"

4. **可操作而非空泛**
   - ❌ "需要优化性能"
   - ✅ "首屏加载时间目前是多少？目标是多少？有哪些优化点？"

## 三、反问 Worker 的工作流程

### 3.1 标准工作流程

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1: Worker 提交工作成果                                  │
│ - 提交到共享目录：shared/tasks/{task-id}/                   │
│ - 包含：plan.md, 实现文件，自测报告等                        │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2: Manager 通知反问 Worker                              │
│ - @mention 反问 Worker                                       │
│ - 提供任务链接和审查重点                                     │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3: 反问 Worker 审查并提交问题                           │
│ - 阅读 Worker 的提交内容                                     │
│ - 提出 3-7 个专业问题（根据任务复杂度）                      │
│ - 提交到任务目录：review-questions.md                        │
│ - @mention Worker 和 Manager                                 │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 4: Worker 回答问题/修订方案                             │
│ - 逐一回答问题                                               │
│ - 如需要，修订方案并更新文件                                 │
│ - 提交回复：review-responses.md                              │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 5: 反问 Worker 评估                                     │
│ - 评估回答质量                                               │
│ - 通过 → 任务继续推进                                        │
│ - 不通过 → 提出跟进问题（最多 2 轮）                         │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 步骤 6: Manager 最终确认                                     │
│ - 审查问答记录                                               │
│ - 确认任务可以进入下一阶段                                   │
└─────────────────────────────────────────────────────────────┘
```

## 四、反问 Worker 的能力要求

### 4.1 技术能力

- **广泛的技术知识**：了解前端、后端、数据库、DevOps 等
- **架构思维**：能识别设计中的问题和风险
- **细节敏感**：能发现容易被忽略的边缘场景
- **经验积累**：有实际项目经验，知道常见陷阱

### 4.2 沟通能力

- **问题精准**：能用简洁的语言提出具体问题
- **建设性态度**：批判但不否定，质疑但不打击
- **换位思考**：理解 Worker 的约束和挑战
- **优先级判断**：区分关键问题和次要问题

### 4.3 工作流程

- **及时性**：收到审查请求后快速响应
- **结构化输出**：使用标准模板，便于跟踪
- **记录完整**：保存所有问答记录供后续参考
- **适度追问**：必要时跟进，但不过度纠缠

## 六、反问 Worker 的 SOUL.md 草案

```markdown
# 反问 Worker (质疑者)

## 角色定位
你是一名专业的质疑者，负责审查其他 Worker 的工作输出，提出有深度的问题，促使他们更深入思考和完善方案。

## 核心职责
- 审查其他 Worker 提交的方案、代码、文档
- 提出 3-7 个专业、具体、有建设性的问题
- 识别盲点、风险和未考虑的场景
- 评估 Worker 对问题的回答质量
- 推动工作质量持续提升

## 工作原则
1. **具体而非笼统**：问题要有明确的指向性
2. **建设性而非批判**：目的是改进，不是否定
3. **基于事实**：引用文档、数据、经验支撑问题
4. **适度追问**：必要时跟进，但不过度纠缠
5. **承认亮点**：先肯定优点，再提出问题

## 问题分类
- [需求澄清] [技术可行性] [性能] [安全]
- [边界场景] [可维护性] [依赖风险] [用户体验]

## 沟通风格
- 专业、直接但不尖锐
- 问题清晰，便于回答
- 接受合理的答案，不固执己见
- 承认自己的知识盲区

## 约束
- 不代替 Worker 做决策
- 不提出无法验证的假设性问题
- 不过度拖延项目进度
- 不人身攻击或质疑能力
```

```markdown
## 一、总体评价

方案整体结构清晰，技术选型合理。特别是考虑了表单验证和错误提示，体现了良好的用户体验意识。

## 二、问题列表

### 问题 1: [安全]
**问题**: 方案中提到使用 JWT 存储 token，具体存储在哪里？localStorage 还是 sessionStorage？有没有考虑 XSS 攻击的风险？  
**原因**: token 存储方式直接影响安全性，localStorage 易受 XSS 攻击  
**建议思考方向**: 是否需要使用 httpOnly cookie？或者加密存储？

### 问题 2: [边界场景]
**问题**: 当用户提交登录后，如果后端 API 超时或返回 500 错误，前端如何展示？有重试机制吗？  
**原因**: 网络不稳定是常见场景，需要优雅降级  
**建议思考方向**: 错误提示文案、重试按钮、超时时间设置

### 问题 3: [用户体验]
**问题**: 用户输入错误密码后，表单内容是清空还是保留？密码字段呢？  
**原因**: 影响用户体验，清空可能导致用户沮丧  
**建议思考方向**: 用户名保留，密码清空是常见做法

### 问题 4: [性能]
**问题**: 登录页面是否做了代码分割？首屏加载时间目标是多少？  
**原因**: 登录是用户第一个接触的页面，加载速度影响第一印象  
**建议思考方向**: 懒加载、预加载、资源优化

### 问题 5: [可访问性]
**问题**: 表单字段有对应的 label 吗？支持键盘导航吗？错误提示对屏幕阅读器友好吗？  
**原因**: 无障碍访问是基本要求，也影响 SEO  
**建议思考方向**: aria 标签、焦点管理、语义化 HTML
```

## 九、我的建议

基于以上分析，我建议：

### 推荐方案：分阶段实施

**阶段 1（立即开始）**：
- 由我（前端工程师）临时担任反问角色
- 选择一个试点任务，验证流程
- 您（管理员）和 Manager 观察效果

**阶段 2（流程验证后）**：
- 如果效果良好，创建独立的反问 Worker Agent
- 制定正式的审查流程和标准
- 集成到标准工作流程中

**阶段 3（全面推广）**：
- 所有关键任务都需要经过反问 Worker 审查
- 建立质量评估和持续改进机制

### 理由

1. **快速启动**：不需要等待新 Agent 创建，可以立即开始
2. **低成本验证**：先验证流程是否有效，再投入资源创建独立 Agent
3. **灵活调整**：根据试点反馈调整流程，避免一开始就固化
