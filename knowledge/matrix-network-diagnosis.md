# Matrix 服务网络连接诊断报告

**诊断时间**: 2026-03-06 21:13 UTC  
**诊断人**: 林小雅（项目总监）

---

## 问题描述

管理员在局域网内尝试连接 Matrix 服务失败：
- ❌ `http://192.168.31.56:18080` - 无法连接
- ❌ `http://matrix-local.hiclaw.io:8080` - 无法连接

---

## 诊断结果

### 1. 服务状态

| 检查项 | 结果 | 说明 |
|--------|------|------|
| Matrix 服务 | ✅ 运行中 | 响应正常 |
| 实际监听地址 | `172.17.0.2:8080` | Docker 容器内部 |
| 容器网络 | `172.17.0.0/16` | Docker bridge 网络 |

### 2. 网络拓扑

### 3. 问题分析

**根本原因**: 
Matrix 服务运行在 Docker 容器内 (`172.17.0.2:8080`)，但 Docker bridge 网络默认**不与局域网直接连通**。

**为什么之前能访问？**
- 通过 `matrix-local.hiclaw.io` 域名解析到 `172.17.0.2`
- 但这只在**同一 Docker 网络**内的容器有效
- 局域网设备无法直接访问 Docker bridge 网络

---

## 解决方案

### 方案 A：端口映射（推荐）⭐

**操作**: 将 Docker 容器的 8080 端口映射到宿主机

```bash
# 1. 找到 Matrix 容器
docker ps | grep matrix

# 2. 重新运行容器，添加端口映射
docker run -d \
  --name matrix-local \
  -p 8080:8080 \
  -p 18080:18080 \
  ...其他配置...
  matrix-image
```

**访问地址**:
```
http://192.168.31.56:8080  # 宿主机 IP + 映射端口
```

**优点**: 
- 简单直接
- 局域网内所有设备都能访问

**缺点**: 
- 需要重启容器

---

### 方案 B：使用 host 网络模式

**操作**: 让容器使用宿主机网络

```bash
docker run -d \
  --name matrix-local \
  --network host \
  ...其他配置...
  matrix-image
```

**访问地址**:
```
http://192.168.31.56:8080  # 直接使用宿主机端口
```

**优点**: 
- 无需端口映射
- 性能更好

**缺点**: 
- 端口可能冲突
- 安全性略低

---

### 方案 C：配置 Docker 网络桥接

**操作**: 配置 Docker 网络与局域网桥接

```bash
# 需要修改 Docker daemon 配置
# /etc/docker/daemon.json
{
  "bip": "192.168.31.1/24",
  "fixed-cidr": "192.168.31.128/25"
}
```

**优点**: 
- 容器获得局域网 IP

**缺点**: 
- 配置复杂
- 需要重启 Docker 服务

---

### 方案 D：临时方案 - 通过 OpenClaw 容器访问

由于 OpenClaw 容器和 Matrix 容器在同一 Docker 网络，可以通过 OpenClaw 转发：

**操作**: 在 OpenClaw 容器内配置反向代理

```bash
# 在 OpenClaw 容器内
nginx -s reload
```

**访问地址**:
```
http://192.168.31.56:18800/matrix  # 假设 OpenClaw 已映射 18800 端口
```

**优点**: 
- 无需重启 Matrix 容器

**缺点**: 
- 需要配置反向代理
- 增加复杂度
