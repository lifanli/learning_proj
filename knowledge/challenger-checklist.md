# Challenger 审查清单

**版本**: v1.0  
**创建时间**: 2026-03-06 13:55 UTC  
**维护人**: Critic Challenger  
**用途**: 按任务类型提供标准化审查要点，帮助 Challenger Worker 快速定位关键问题

---

## 一、使用说明

### 1.1 如何使用本清单

1. **识别任务类型** → 选择对应的审查清单
2. **逐项检查** → 对每个要点提出具体问题
3. **问题分级** → 按🔴🟡🟢分类（见 1.2 节）
4. **输出报告** → 使用标准审查报告模板

### 1.2 问题分级标准

| 级别 | 标识 | 定义 | 处理要求 | 示例 |
|------|------|------|----------|------|
| **阻塞级** | 🔴 | 可能导致严重故障、安全漏洞、重大返工 | 必须解决才能继续 | "这个 API 没有鉴权机制" |
| **建议级** | 🟡 | 可能影响质量、性能、可维护性 | 强烈建议优化 | "建议添加单元测试覆盖边界情况" |
| **参考级** | 🟢 | 可供未来迭代参考的改进点 | 记录在案，择机处理 | "未来可以考虑引入 TypeScript" |

### 1.3 审查深度标准

| 任务级别 | 描述 | 审查要求 | 问题数量参考 |
|----------|------|----------|--------------|
| **S 级** | 核心功能、高风险 | 强制审查，逐项检查 | 5-7 个 |
| **A 级** | 重要功能、中等风险 | 强制审查，重点检查 | 3-5 个 |
| **B 级** | 常规功能、低风险 | 可选审查 | 2-3 个 |
| **C 级** | 小修复、文档更新 | 免审查 | - |

## 二、通用审查维度（所有任务类型）

无论什么任务类型，都应检查以下维度：

| 维度 | 审查要点 | 典型问题 |
|------|----------|----------|
| **需求理解** | 是否准确理解任务目标？ | "核心用户场景是什么？边缘情况考虑了吗？" |
| **技术方案** | 方案是否可行？有无技术难点？ | "这个方案的技术风险在哪里？有备选方案吗？" |
| **安全性** | 有无安全风险？ | "有没有考虑 XSS/SQL 注入/未授权访问？" |
| **性能** | 有无性能瓶颈？ | "预期 QPS 是多少？有压测数据吗？" |
| **可维护性** | 代码/方案是否易于维护？ | "有文档吗？新人能看懂吗？" |
| **可测试性** | 是否易于测试？ | "有单元测试吗？测试覆盖率多少？" |
| **监控告警** | 如何发现问题？ | "有监控指标吗？告警阈值怎么设？" |
| **回滚方案** | 出问题如何恢复？ | "有回滚方案吗？RTO 是多少？" |

## 三、前端任务审查清单

### 3.1 需求与设计

- [ ] 核心用户场景是否明确？
- [ ] 边缘情况是否考虑（空状态、加载失败、网络超时）？
- [ ] 设计稿是否完整（所有状态：默认/加载/成功/错误）？
- [ ] 可访问性是否考虑（WCAG 标准、键盘导航、屏幕阅读器）？

### 3.2 技术选型

- [ ] 为什么选 A 框架不选 B？（React vs Vue vs Angular）
- [ ] 组件库选型理由？（Ant Design vs MUI vs 其他）
- [ ] 状态管理方案是否必要？（复杂度高才需要）
- [ ] 构建工具选型？（Vite vs Webpack）

### 3.3 性能优化

- [ ] 首屏加载时间目标是多少？
- [ ] 是否做代码分割（Code Splitting）？
- [ ] Bundle 大小预估？有无优化计划？
- [ ] 图片/资源是否优化（懒加载、压缩、CDN）？
- [ ] 有无性能监控方案（Web Vitals）？

### 3.4 兼容性

- [ ] 支持哪些浏览器？（Chrome/Firefox/Safari/Edge）
- [ ] 是否支持移动端？
- [ ] 降级方案是什么？（不支持的特性如何优雅降级）

### 3.5 安全

- [ ] 用户输入是否校验？（XSS 防护）
- [ ] Token 存储方式？（localStorage vs httpOnly cookie）
- [ ] 敏感信息是否加密？
- [ ] CSRF 防护是否考虑？

### 3.6 测试

- [ ] 单元测试覆盖率目标？
- [ ] E2E 测试是否覆盖核心流程？
- [ ] 是否有视觉回归测试？

## 四、后端任务审查清单

### 4.1 API 设计

- [ ] REST 还是 GraphQL？理由是什么？
- [ ] API 版本管理策略？
- [ ] 请求/响应格式是否统一？
- [ ] 错误码设计是否规范？
- [ ] 接口文档是否完整（Swagger/OpenAPI）？

### 4.2 数据库

- [ ] 数据库选型理由？（MySQL vs PostgreSQL vs MongoDB）
- [ ] 索引策略是什么？
- [ ] 查询性能如何保证？（慢查询分析）
- [ ] 是否需要分库分表？触发条件是什么？
- [ ] 数据迁移方案？

### 4.3 缓存

- [ ] 缓存策略？（Redis/Memcached/本地缓存）
- [ ] 缓存失效机制？
- [ ] 缓存穿透/击穿/雪崩如何防护？
- [ ] 缓存一致性如何保证？

### 4.4 安全

- [ ] 身份认证方案？（JWT/OAuth2/Session）
- [ ] 权限控制粒度？（RBAC/ABAC）
- [ ] SQL 注入防护？（参数化查询）
- [ ] 敏感数据是否加密存储？
- [ ] 日志是否脱敏？

### 4.5 性能与扩展

- [ ] 预期 QPS 是多少？
- [ ] 水平扩展方案？
- [ ] 数据库读写分离是否考虑？
- [ ] 消息队列是否必要？（异步处理）
- [ ] 限流熔断方案？

### 4.6 监控与日志

- [ ] 核心监控指标？（延迟/错误率/QPS）
- [ ] 日志收集方案？（ELK/云厂商方案）
- [ ] 链路追踪是否考虑？（Jaeger/Zipkin）
- [ ] 告警阈值如何设定？

## 五、DevOps 任务审查清单

### 5.1 架构设计

- [ ] 有无单点故障？
- [ ] 故障转移机制？
- [ ] 高可用方案？（多可用区/多地域）
- [ ] 负载均衡策略？

### 5.2 容量规划

- [ ] 峰值预估依据？（历史数据/业务预期）
- [ ] 压测数据支撑？
- [ ] 弹性伸缩策略？（自动扩缩容条件）

### 5.3 CI/CD

- [ ] 构建流程是否自动化？
- [ ] 测试门禁？（Lint/单元测试/集成测试）
- [ ] 部署策略？（蓝绿/金丝雀/滚动）
- [ ] 回滚流程是否验证过？

### 5.4 安全合规

- [ ] 最小权限原则？（IAM 策略）
- [ ] 密钥管理方案？（Vault/云厂商 KMS）
- [ ] 数据加密？（传输中/静态）
- [ ] 合规要求？（GDPR/等保）

### 5.5 监控告警

- [ ] 基础设施监控？（CPU/内存/磁盘/网络）
- [ ] 应用监控？（延迟/错误率/饱和度）
- [ ] 告警分级？（P0/P1/P2）
- [ ] 告警通知渠道？（短信/电话/IM）
- [ ] 告警疲劳如何避免？

### 5.6 灾备恢复

- [ ] 备份策略？（频率/保留期）
- [ ] 备份验证？（定期恢复演练）
- [ ] RTO（恢复时间目标）？
- [ ] RPO（恢复点目标）？
- [ ] 灾难恢复预案？

### 5.7 成本控制

- [ ] 月度成本估算？
- [ ] 成本优化空间？（预留实例/Spot 实例）
- [ ] 成本告警阈值？

## 九、高频问题库（持续更新）

### 9.1 安全类（出现频率最高）

| 问题 | 适用场景 |
|------|----------|
| "有没有考虑 XSS/SQL 注入防护？" | 前端表单、后端 API |
| "Token 存储在哪里？是否考虑 httpOnly cookie？" | 用户认证 |
| "敏感数据是否加密存储？" | 用户信息、支付数据 |
| "有没有最小权限原则？" | IAM、数据库权限 |

### 9.2 性能类

| 问题 | 适用场景 |
|------|----------|
| "预期 QPS 是多少？有压测数据吗？" | 后端 API、数据库 |
| "首屏加载时间目标是多少？" | 前端页面 |
| "有没有做缓存？缓存失效机制？" | 数据查询、API 响应 |
| "Bundle 大小预估？" | 前端构建 |

### 9.3 可维护性类

| 问题 | 适用场景 |
|------|----------|
| "有文档吗？新人能看懂吗？" | 所有任务 |
| "有单元测试吗？覆盖率多少？" | 代码开发 |
| "日志是否完整？便于排查问题吗？" | 后端服务 |
| "配置和代码是否分离？" | 所有任务 |
