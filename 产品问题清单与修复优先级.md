# 产品问题清单与修复优先级

## 1. 文档目的

这份文档基于当前代码仓库的真实实现，从产品视角整理项目当前存在的关键问题。

目标不是讨论抽象架构，而是回答更直接的问题：

1. 这个产品现在最可能在哪些地方让用户困惑
2. 哪些问题会直接影响真实使用体验
3. 哪些问题会影响结果可信度
4. 哪些问题应该最优先修复

本文档按“问题 -> 证据 -> 用户影响 -> 严重级别 -> 修复建议”的方式整理，便于后续直接转成改造计划。

---

## 2. 产品一句话定位

当前项目本质上是一个：

“AI 自主学习、素材沉淀、自动技术出版”的知识生产系统。

目标链路大致是：

学习目标
-> 课程规划
-> 自动找资源与学习
-> 沉淀素材库
-> 自动组织与出版
-> 输出本地 Markdown 知识库

这个定位是成立的，且有明显产品潜力。

但当前项目的主要问题，不是功能缺失，而是：

- 主路径不够清晰
- 用户对结果可信度缺少稳定预期
- 核心中台价值尚未被产品化表达
- 新旧系统并存导致叙事不统一

---

## 4. 详细问题清单

## 问题 1：产品主线不清晰，用户不知道应该怎么开始

### 现象

当前系统中存在较多入口与运行模式，用户可能无法快速判断最推荐的使用路径。

### 证据

CLI 中同时存在大量模式：

- `daily`
- `wechat`
- `arxiv`
- `doc`
- `course`
- `queue`
- `resume`
- `status`
- `study-course`
- `study-wechat`
- `study-github`
- `study-arxiv`
- `study-topic`
- `publish`
- `full`
- `curriculum`
- `auto-study`
- `materials`

相关文件：

- `main.py`
- `app.py`

### 用户影响

用户容易产生这些疑问：

- 我应该先生成课程表，还是直接开始抓资源？
- `study-topic` 和 `curriculum` 的关系是什么？
- `publish` 和 `full` 分别适合什么场景？
- 旧模式和新模式有何差异？
- 如果我只是想做一条完整学习-出版链路，到底应该点哪里？

这种问题的直接结果是：

产品虽然功能很多，但用户缺少“默认主旅程”。

### 严重程度

高（P0）

### 修复建议

1. 明确唯一推荐主路径：

   `目标 -> 课程表 -> 学习 -> 素材 -> 出版 -> 浏览成品`

2. 在 UI 首页明确提供“推荐流程”入口，而不是并列展示大量功能页。
3. 在 CLI 中区分：
   - 主线模式
   - 高级/调试/兼容模式
4. 给每个入口模式增加一句“适合什么场景”的说明。
5. 将非主线模式归入“高级工具”或“兼容模式”组。

---

## 问题 2：新旧架构并存，产品叙事不统一

### 现象

仓库中同时存在两代系统：

旧系统：
- `src/graph/pipeline.py`
- `src/agents/editor_agent.py`
- `src/storage/rag_manager.py`

新系统：
- `src/student/`
- `src/publisher_v2/`
- `src/core/material_store.py`

### 证据

旧系统更偏：
- 单条内容处理
- RAG 驱动的单篇文章生成

新系统更偏：
- 多来源学习
- 素材库中台
- 自动成书

二者同时存在，且都仍在入口层可达。

### 用户影响

会导致产品定位模糊：

- 这是一个单文知识整理器？
- 还是一个 AI 学习出版系统？

同时也会影响协作者和你自己后续迭代判断：

- 新功能应该挂旧链路还是新链路？
- 哪一套才是产品未来主线？

### 严重程度

高（P0）

### 修复建议

1. 正式定义：`StudentAgent + MaterialStore + PublisherAgent` 为唯一主架构。
2. 旧 `KnowledgePipeline` 标记为：
   - legacy
   - compatibility
   - single-article quick flow
3. 文档统一改写，避免继续平行叙事。
4. UI/CLI 中将 legacy 功能明确隔离，不与主线混排。

---

## 问题 3：出版质量控制存在，但质量闭环不完整

### 现象

当前出版链已经包含质量审核，但更多是“发现问题并打标”，还不是“发现问题后自动闭环修复”。

### 证据

相关链路：

- `src/publisher_v2/publisher_agent.py`
- `src/publisher_v2/workers/quality_reviewer.py`
- `src/publisher_v2/workers/section_writer.py`

当前流程大致为：

1. 写章节
2. 代码注释/图片整合/交叉引用
3. 质量审核
4. 如果有问题，加入 warnings 或 flags
5. 输出到最终成品中

`QualityReviewer` 当前更偏：
- 评分
- 问题说明
- 建议输出

而不是：
- 自动返工
- 自动重写
- 自动回炉再审

另外，审核对长内容是采样，不是全文逐段审核。

### 用户影响

用户会担心：

- 这本书虽然生成出来了，但真的可靠吗？
- 标了 warning 的章节是否应该重新写？
- 系统是否只是“把问题写进报告”，而不是解决问题？

最终结果是：

产品能产出内容，但用户未必愿意把它当成“可放心阅读的成品”。

### 严重程度

高（P0）

### 修复建议

1. 对关键质量问题增加自动返工机制，例如：
   - too_short
   - review_failed
   - 明显结构不完整
2. 把出版链升级成：
   - 写作
   - 质检
   - 返工
   - 再审
   - 再输出
3. 在最终成品中区分：
   - 已通过质量门槛
   - 可读但待改进
   - 不建议作为最终版本
4. 在 UI 中把质量状态可视化，而不只是写在 report.json 里。

## 问题 4：素材库已经是核心中台，但对用户的价值不够显性

### 现象

`MaterialStore` 的设计非常重要，但当前它更像系统内部资产仓库，而不是用户可操作、可判断的知识工作台。

### 证据

相关文件：

- `src/core/material_store.py`
- `app.py`

素材包含：
- source_url
- source_type
- content
- summary
- tags
- images
- code_blocks
- references
- terms
- metadata

但当前缺少显式面向用户的字段或视图来回答：

- 这条素材质量高不高？
- 是否已经翻译完成？
- 是否适合用于出版？
- 是否与其他素材重复？
- 覆盖了当前主题的哪一部分？

### 用户影响

用户能看到“素材存在”，但不一定能做判断。

这意味着：
- 无法放心选材
- 无法知道素材库沉淀是否真的有价值
- 素材多了之后会越来越难管

### 严重程度

中高（P1）

### 修复建议

给 `Material` 增加更明确的质量元数据，例如：

- `quality_score`
- `completeness_score`
- `translation_status`
- `ready_for_publish`
- `source_reliability`
- `review_status`

同时在 UI 中展示：
- 可出版素材
- 待补全素材
- 重复或噪声素材
- 高价值素材排行

## 问题 5：资源发现能力已经比较强，但资源筛选能力还不够强

### 现象

`TopicExplorer` 已经能从多种来源找资源，但目前“找到资源”强于“筛出最适合当前学习阶段的资源”。

### 证据

相关文件：

- `src/student/topic_explorer.py`
- `src/student/student_agent.py`

当前渠道：
- registry
- web
- github
- arxiv

当前排序主要依赖：
- registry 命中
- web 搜索结果
- GitHub stars
- ArXiv 搜索结果
- relevance_score 简单排序

### 用户影响

会出现这些风险：

- 找到很多资源，但其中一些不够适合当前目标
- 资源之间可能重复
- 学习路径可能噪音偏高
- 不同难度阶段的用户拿到相似结果

也就是说，系统显得很主动，但不一定真的“教得好”。

### 严重程度

中高（P1）

### 修复建议

1. 在资源排序里加入更多维度：
   - 难度匹配
   - 来源可信度
   - 互补性
   - 是否与已学内容重复
2. 结合 curriculum topic 的 difficulty 进行筛选。
3. 对资源增加“推荐理由”字段，告诉用户为什么选它。
4. 区分：
   - 入门资源
   - 深入资源
   - 工程实践资源
   - 前沿研究资源

---

## 问题 6：课程规划能力是亮点，但默认体验还不够强

### 现象

`CurriculumAgent` 是非常有产品价值的能力，但当前仓库默认状态下，它的成果感不够强。

### 证据

相关文件：

- `src/student/curriculum_agent.py`
- `config/curriculum.yaml`

当前 `config/curriculum.yaml` 内容很短，只是：
- goal
- depth
- status: draft
- domains: []

这意味着课程表能力虽然存在，但默认仓库状态不能直接体现它的完整价值。

### 用户影响

用户可能知道“有自动课程规划”，但第一次接触时未必能感受到：
- 系统真的给我规划了完整学习路径
- 不同主题之间真的有层次关系
- 这个能力比简单搜索更有价值

### 严重程度

中（P1）

### 修复建议

1. 提供一个真实可展示的课程表示例。
2. 在 UI 中把课程表作为主入口之一，而不是埋在功能列表里。
3. 给课程表展示：
   - 主题数
   - 完成进度
   - 每个主题对应的学习资源
4. 对 draft/approved/completed 状态给出更明显的产品反馈。

---

## 问题 7：配置、依赖与运行行为存在漂移风险

### 现象

当前配置和依赖之间存在不完全一致的问题，可能导致实际运行行为与预期不一致。

### 证据

`config/settings.yaml` 当前配置：
- provider: anthropic
- model: claude-opus-4-6
- base_url: `https://cc.580ai.net`

但 `requirements.txt` 中未看到 `anthropic` 依赖。

而 `src/core/llm_client.py` 中存在逻辑：
- 如果 anthropic SDK 未安装，则自动回退到 openai provider 路径

### 用户影响

这类问题的危险在于它可能不是显式报错，而是“悄悄走了另一条兼容路径”。

后果包括：
- 配置看起来是 Claude，实际行为不一定完全一致
- 同一套配置在不同环境表现不同
- 调试困难
- 用户难以建立稳定预期

### 严重程度

中高（P1）

### 修复建议

1. 让配置与依赖强一致。
2. 如果当前默认走 anthropic，就把 `anthropic` 加进依赖清单。
3. 对 provider fallback 给出显式日志和 UI 告警，不要 silent fallback。
4. 增加“当前实际模型提供方/调用模式”的运行时展示。

## 问题 8：文档与真实系统状态存在漂移

### 现象

项目文档中的系统描述与当前真实实现已经不完全一致。

### 证据

相关文件：
- `架构设计.md`
- `config/settings.yaml`
- `src/student/*`
- `src/publisher_v2/*`

文档中更偏旧叙事：
- Qwen / DashScope / LightRAG 为主中心
- 旧角色分工更突出

但真实系统已经更明显往：
- Student / Publisher 双智能体
- MaterialStore 中台
- Anthropic 兼容配置
- 成书型输出

方向演进。

### 用户影响

会导致：
- 协作者理解偏差
- 产品定位表述混乱
- 用户对能力预期不一致
- 后续迭代容易被过时叙事绑住

### 严重程度

中（P1）

### 修复建议

1. 重写顶层架构文档，明确新主线。
2. 将旧体系说明单独标成 legacy。
3. 所有“如何使用项目”的文档都围绕新主旅程重写。

## 问题 9：入口层文件过大，产品迭代效率会持续下降

### 现象

当前两个入口文件体积已经较大：

- `app.py`：1578 行
- `main.py`：604 行

### 证据

这说明 UI 状态、页面逻辑、任务入口逻辑正在持续堆积在顶层入口文件里。

### 用户影响

对用户不是立刻可见的问题，但会很快变成产品迭代问题：

- 加一个功能容易影响别的页面
- 页面行为不稳定风险增加
- 入口逻辑越来越难维护
- 产品优化速度会逐步下降

### 严重程度

中（P2）

### 修复建议

1. 拆分 `app.py`：
   - pages/
   - services/
   - state helpers/
2. 拆分 `main.py`：
   - cli handlers/
   - command routers/
3. 入口文件只保留路由与初始化，不继续承载业务细节。

## 问题 10：任务与质量状态的可观测性还不够强

### 现象

系统内部其实有不少状态与报告，但还没有完全转化成面向用户的“过程透明度”。

### 证据

已有能力：
- Task DAG checkpoint
- publish_report
- curriculum progress
- log system

但缺少统一的、面向产品的状态视图，例如：
- 当前学习到哪一步
- 哪些素材质量高
- 哪些章节因为质量问题待返工
- 哪些主题资料还不足

### 用户影响

用户会产生这些感受：
- 系统看起来在跑，但不知道现在做得怎么样
- 失败或质量问题要靠日志和 report 才能发现
- 很难快速做运营或人工接管决策

### 严重程度

中（P2）

### 修复建议

建立统一“可观测面板”概念，至少覆盖：

1. 学习状态
   - 当前主题
   - 已抓取资源数
   - 成功率/失败率
2. 素材状态
   - 高质量素材
   - 待补全素材
   - 重复素材
3. 出版状态
   - 已完成章节
   - 审核失败章节
   - 待返工章节

## 问题 11：素材处理职责边界不够显式，后续容易越长越乱

### 现象

当前系统已经具备素材加工能力，但“提取正文、提取代码、提取图片、提取术语、找引用、翻译、打标签”这几件事，在产品和架构层面的边界还不够显式。

换句话说，系统已经在做这些事，但对外和对内都还没有把它们清晰定义成一条模块化流水线。

### 证据

相关代码主要集中在：

- `src/student/student_agent.py`
- `src/student/workers/`
- `src/core/task_engine.py`
- `src/core/material_store.py`

当前 `StudentAgent` 已经有 enrich DAG，但从产品理解层面，这一段仍然容易被看成一个笼统的“加工步骤”，而不是明确的三层模块：

- 基础提取层
- 语义加工层
- 质量整理层

### 用户影响

短期内，这个问题未必直接表现为“不能用”；
但中期会明显影响产品演进：

- 某个处理能力升级时，容易牵动一大片逻辑
- 一个步骤失败时，不容易优雅降级
- 素材质量会受局部逻辑耦合影响，稳定性变差
- 后续做素材质量评分、出版可用性判断时，边界会越来越模糊

最终结果是：

系统会越来越强，但也越来越难维护、难解释、难保证稳定输出。

### 严重程度

中高（P1）

### 修复建议

把素材处理正式建模为独立流水线，而不是“StudentAgent 内部的一团处理逻辑”。

建议明确三层：

1. 基础提取层
   - 正文提取
   - 代码提取
   - 图片提取
   - 引用提取

2. 语义加工层
   - 术语提取
   - 翻译
   - 标签生成
   - 摘要生成（可选）

3. 质量整理层
   - 去重
   - 完整性检查
   - 质量评分
   - 出版可用性标记

目标是把当前：

`抓取 -> 一次性处理 -> 存储`

升级为：

`抓取 -> 多模块加工 -> 质量整理 -> 存储`

这样 Material 才会真正成为稳定的知识资产，而不是一次性中间产物。

## 5. 对用户体验影响最大的四个问题

如果从“会最直接影响真实用户体验”的角度排序，我认为最关键的是这四个：

### 1. 不知道怎么开始
对应问题：主线不清晰

### 2. 不知道结果是否可靠
对应问题：出版质量闭环不完整

### 3. 不知道素材库到底有没有价值
对应问题：素材中台对用户不可见

### 4. 不知道系统当前真正运行在什么状态
对应问题：可观测性不足、配置行为漂移

---

## 6. 修复优先级建议

## P0：立即优先解决

### P0-1 统一主产品路径
目标：让用户不用思考“该走哪条链路”。

## 8. 总结

当前产品最核心的结论是：

这个项目已经不是“功能型原型”，而是一个明确面向知识生产的系统雏形。

它的长板很清楚：
- 学习链路完整
- 素材中台已经成型
- 自动出版能力已具备雏形

但它最需要修的，不是继续堆新功能，而是把产品路径和质量预期收束起来。

### 一句话总结

当前项目最大的产品问题，不是“不会做”，而是“会做很多，但还没有把默认用户路径、结果可信度和中台价值表达成一个清晰稳定的产品体验”。
